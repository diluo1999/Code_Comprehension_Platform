Reference Guide

Naming Conventions

Providing concise, relevant variable and function names is one of the most important features of good, readable, and maintainable code.

Python naming conventions discourage long identifiers. This function's name is unnecessarily long; we know it will add two numbers by the nature of its parameters. We can easily reduce it. 
def thisAddsTwoNumbers(number1, number2):
    return number1 + number2
Instead, consider the following: 
def add(n1, n2):
    return n1 + n2
In Python, most educators and developers adhere to a convention called "snake case" for variable and function names, in which all words are lowercase and separated by underscores. Using other naming conventions, such as camelcase/mixedcase (capitalizing the first letter of every word after the first word) is not recommended in Python. 
vegetablesPicked = 17
vegetablesRemaining = 9
Using snake case, we can ensure our Python code is neat and consistent, matching any modules we may also incorporate. 
apples_eaten = 4
bananas_eaten = 7
Unlike variable and function identifiers, Python user-defined class names should follow the capwords naming convention (the first letter of each word should be capitalized): 
class CountApples(Apple):

Code Layout

Surround top-level function and class definitions with two blank lines.
Method definitions inside a class are surrounded by a single blank line. 
class Dog:
  def __init__(self, name):
    self.name = name
    self.tricks = []

  def add_trick(self, trick):
    self.tricks.append(trick)


class Cat:
  def __init__(self, name):
    self.name = name
Imports should usually be on separate lines: 
import sys, os
import os
import sys
It’s okay to say this though: 
from subprocess import Popen, PIPE
Operators sit far away from their operands: 
income = (gross_wages +
          taxable_interest +
          (dividends - qualified_dividends) -
          ira_deduction -
          student_loan_interest)
Easy to match operators with operands: 
income = (gross_wages
          + taxable_interest
          + (dividends - qualified_dividends)
          - ira_deduction
          - student_loan_interest)

Whitespace

Avoid whitespace immediately inside parentheses, brackets or braces.

Wrong: 
spam( ham[ 1 ], { eggs: 2 } )
Correct: 
spam(ham[1], {eggs: 2})

Avoid whitespace between a trailing comma and a following close parenthesis.

Wrong: 
bar = (0, )
Correct: 
foo = (0,)
  • Consider adding whitespace around the operators with the lowest priority(ies).

    • Wrong: 
    hypot2 = x * x + y * y
c = (a + b) * (a - b)
    Correct: 
    hypot2 = x*x + y*y
c = (a+b) * (a-b)

  • Never use more than one space.

    • Wrong: 
    i  =  i  +  1
    Correct: 
    i = i + 1

  • Always have the same amount of whitespace on both sides of a binary operator.

    • Wrong: 
    submitted +=1
    Correct: 
    submitted += 1

Linguistic Antipatterns

Although LAs will not result in errors or make a program fail to compile, they are considered poor-practice and make programs less readable and harder to maintain.

When we are writing "setter" methods, we don't want them to return anything; they should simply set a property to the provided parameter. A common linguistic antipattern occurs when there is anything returned in this method, which should always be void. 
def set_name(self, name):
    self.name = name
    return self.name
A good "set" method simply sets the parameter and does not return anything: 
def set_location(self, location):
    self.location = location
"Get" methods should always be simple, one-line functions that return the requested value and nothing more. A bad "getter" performs other operations in addition to this. 
def get_count(self):
    self.count += 1
    if self.count > 10:
        self.count = 0
    return self.count
When writing a "getter," it should be solely an accessor: 
def get_id(self):
    return self.id
"Get" methods should always return something. If a "getter" is void, it can't help us access the variable we're requesting, which will create errors later. 
def get_video_url(self):
    video_url = self.url
A proper "getter" has a return statement for the variable requested: 
def get_dog_breeds(self):
    return self.breed_list
A method with "is" in its identifier implies that it will return a boolean value. If it returns a value that is not True or False, it is incorrectly named. The following method returns a number instead of a boolean: 
def is_alive(self):
    return self.age
Proper "is" methods will return True, False, or a boolean variable: 
def is_adult(self):
    if self.age > 18:
        return True
    return False
The name of a method is in the form of predicate whereas the return type is not Boolean.

Example: method is_alive does not return. 
def is_alive(self):
    if self.status == 0:
        self.status = 1
Make the method return a Boolean: 
def is_alive(self):
    if self.status == 0:
      return True
    else:
      return False
The documentation of a method is in contradiction with its declaration.

Example: method is_alive is in contradiction with its comment documenting “is dead”, as “alive” and “dead” are antonyms. 
def is_alive(self):
      ''' returns True if the cell is dead '''
      if self.status == 0:
        return True
      else:
        return False
Make method name and docstring consistent: 
def is_alive(self):
      ''' returns True if the cell is alive '''
      if self.status == 0:
        return True
      else:
        return False
